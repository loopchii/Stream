# Privacy and Data Use

Stream is a public research repository. This document explains the boundary of the data it uses, what it does not do, and how contributors should think about attribution, privacy, and reuse.

This document describes the current repository posture. It is not legal advice.

## What Stream uses today

Stream currently operates across three distinct data lanes:

| Lane | What it contains | Why it exists |
|---|---|---|
| Synthetic representation lane | Deterministic synthetic rows generated by the repository itself | Method teaching, testing, and reproducible UI behavior without pretending to measure a real catalogue |
| Public music lane | Publicly available music-video metadata committed in `data_sources/` and processed into `data/music/` | Real-world pressure testing for the music, attention, and virality analysis surfaces |
| Reference-link lane | Search links and enrichment references to public sources such as MusicBrainz, Wikipedia, IMSLP, MuseScore, Ultimate Guitar, and AllMusic | Helps contributors extend the music intelligence layer without bundling third-party databases into the repo |

## What Stream does not do

Stream is intentionally narrow.

It does not:

- collect private user data
- track end users across websites or devices
- store account passwords, payment information, or private platform exports
- republish raw audiovisual content from third-party services
- claim ownership over third-party datasets or metadata
- function as a hidden commercial surveillance tool

## Public music data boundary

The current music lane is built from public metadata and committed CSV inputs in `data_sources/`, including public video titles, channel names, view counts, durations, tags, descriptions, and date fields where available.

Those inputs are used to measure patterns such as:

- concentration of attention
- publication-year coverage
- genre visibility
- collaboration share
- repeatability and drift proxies

The repository analyzes those public fields. It does not mirror video files, audio files, or private platform analytics.

## Attribution and provenance

If a dataset or reference source materially contributes to the repository, it should be documented in [../data/README.md](../data/README.md).

At minimum, that entry should state:

- source name
- access path or homepage
- whether the repository stores the raw data, a transformed derivative, or only a reference link
- license or reuse posture if known
- what analysis surface depends on it

If the source license is unclear, the contribution should pause until the boundary is clear.

## Privacy posture

Stream is designed around public or synthetic data, not personal dossiers.

That means:

- the synthetic representation lane contains no real people
- the music lane is evaluated as public metadata, not as a private behavioral profile
- the repository is not intended to process sensitive personal data
- conclusions should stay at the level of patterns, structures, and representation, not individual judgment

If a contributor wants to add a dataset that contains personal information, special-category data, or scraped private content, that work should not be merged into this repository without a separate review and a rewritten data policy.

## GDPR and CCPA posture

This repository is not presented as a consumer data product, a targeted advertising system, or a personal data brokerage surface.

Its current posture is:

- public research and analysis
- synthetic teaching and testing data
- public metadata analysis
- no intentional collection of private end-user data

That does not remove the need for care. It means the right discipline here is to keep the repository narrow, well-attributed, and honest about what it is actually processing.

## Fair-use and reuse posture

Stream is built for analysis, criticism, teaching, and inspectable research. That is the purpose of the repository.

The repository should stay on the safe side of that posture by:

- analyzing patterns instead of reproducing expressive works
- citing source systems instead of pretending their data is original to Stream
- publishing derived metrics and summaries instead of redistributing large third-party raw corpora without permission
- keeping claim boundaries visible when the source material is thin or mixed

## Contributor rules for new data

Before adding or refreshing a dataset, contributors should be able to answer all of the following:

1. Where did the data come from?
2. Is the source public, synthetic, licensed, or unclear?
3. Are we storing raw data, transformed data, or just derived metrics?
4. Does the addition change a public claim?
5. Does the README or data documentation need to change alongside the code?

If those questions are not answerable, the data is not ready to merge.

## If you need to raise a concern

If you think a data source, claim, or attribution line is wrong:

- open an issue with the label `data-boundary` or `docs`
- point to the exact file or surface
- explain whether the problem is provenance, licensing, privacy, or overclaiming

The goal is to correct the public surface quickly and visibly.
